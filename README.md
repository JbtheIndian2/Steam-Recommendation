# Adjacent — a content-based Steam game recommender

A recommendation engine that suggests games based on **what a game is about**
— its genres, community tags, and play modes — rather than on other users'
behavior. Given a game, it returns its nearest neighbors in a learned
tag-vector space.

Built as a portfolio project to demonstrate a full, small ML pipeline:
raw data → cleaning → feature engineering → a real scikit-learn model →
a served API → a UI on top of it.

## Live demo

```
python app.py
# open http://localhost:5000
```

## Why content-based (not collaborative filtering)

Collaborative filtering (the "users who liked X also liked Y" approach)
needs a user-item interaction matrix, which this dataset doesn't have —
just store-page metadata and aggregate review counts. Content-based
filtering works with exactly that: it doesn't need a single user account,
recommends cold-start games (no rating history required beyond a minimum
review count), and is fully explainable — every recommendation comes with
the specific tags that drove it.

The tradeoff: it can't learn "people who like A also happen to like B" if B
doesn't *look* like A on paper. That's the natural next extension — see
[Limitations](#limitations--next-steps).

## Methodology

**1. Cleaning & filtering** (`src/preprocess.py`)
Raw dataset: 27,075 titles. Filtered to English-language entries with
≥100 combined positive/negative reviews, which removes asset-flips and
shovelware that would otherwise dilute the tag vocabulary. Left with
**8,640 games**.

**2. Feature engineering — the "soup"**
Each game's genres, Steam-community tags, and a curated subset of
categories (the ones that signal play style — `Co-op`, `Single-player`,
`MMO`, `VR Support`, etc. — not store boilerplate like `Steam Achievements`)
are combined into one namespaced token list, e.g.:

```
genre__Action|genre__Adventure|tag__Puzzle|tag__Co-op|cat__Single-player
```

Namespacing (`genre__` / `tag__` / `cat__`) keeps identical words in
different fields — genre "Action" vs. tag "Action" — as distinct signals,
since they carry different information (a genre is Valve's own
classification; a tag is crowd-sourced).

**3. Vectorization — `sklearn.feature_extraction.text.TfidfVectorizer`**
A custom tokenizer splits on `|` so multi-word tokens like
`tag__Turn-Based Strategy` survive as one unit instead of being shredded by
whitespace tokenization. TF-IDF then downweights common tokens (`Indie`
appears on ~40% of games and says little) and upweights rare, descriptive
ones (`Metroidvania`, `Bullet Hell`) automatically via:

```
idf(t) = ln((1 + N) / (1 + df(t))) + 1
```

Result: an **8,640 × 310** sparse matrix, ~97.6% sparse (each game only
has ~7 tokens out of 310 possible).

**4. Indexing — `sklearn.neighbors.NearestNeighbors`**
Fit with `metric='cosine'` over the TF-IDF matrix. Cosine similarity on
L2-normalized TF-IDF vectors reduces to a dot product of shared, weighted
tokens — exactly the "how much taste-overlap do these two games have"
question this project is trying to answer.

**5. Explaining a match**
For any pair of games, the element-wise product of their TF-IDF vectors
gives each shared token's contribution to the similarity score. The
highest-contributing tokens become the "shared: X, Y, Z" explanation shown
in the UI — the recommendation is never a black box.

## Setup

```bash
pip install -r requirements.txt

# place the raw Kaggle "Steam Store Games" CSV at data/steam.csv
# https://www.kaggle.com/datasets/nikdavis/steam-store-games

python src/preprocess.py     # -> data/games_clean.csv
python src/train.py          # -> models/*.joblib, *.npz, *.csv

python app.py                # serves the UI + API at localhost:5000
# or, no UI needed:
python src/recommend.py "Half-Life 2"
```

## Limitations & next steps

- **No collaborative signal.** Two games with identical tags but wildly
  different actual player sentiment ("feels like X" vs. "is tagged like X")
  are treated as equally similar. Blending in playtime or ownership data as
  a secondary signal would help.
- **Tags are Steam's crowd-sourced folksonomy**, not a ground-truth
  taxonomy — they're noisy and skew toward whatever early reviewers typed.
- **Text description isn't used.** Steam's short/long descriptions
  (unused here) would let a semantic-embedding model (e.g.
  `sentence-transformers`) catch similarity that tags miss — worth an
  ablation to compare against this TF-IDF baseline.
- **No offline evaluation metric.** This project uses qualitative
  spot-checking, not something like held-out precision@k. A good extension:
  treat "same series" or "same tag cluster" as a weak ground truth and
  measure how often the model recovers it.
